## Context

현재 코드베이스는 Django 기본 스캐폴드 수준으로 도메인 생성 API/서비스가 아직 구체화되지 않았다. 이번 변경은 향후 도메인 생성 플로우에서 중복 체크와 자동 추천을 기본 동작으로 만들기 위한 첫 구현이며, 서버 사이드(HTMX/일반 요청 공용)에서 일관된 결과 구조를 제공해야 한다.

## Goals / Non-Goals

**Goals:**
- 입력 후보 리스트를 받아 중복(대소문자/하이픈 정규화 기준) 여부를 판별한다.
- 각 후보의 사용 가능 여부를 확인하고 사용 불가 시 대체 추천을 생성한다.
- 뷰 레이어에서 바로 사용할 수 있는 표준 응답 구조(`candidate`, `status`, `reason`, `alternatives`)를 제공한다.
- 결정적 테스트 더블을 사용해 네트워크 의존성 없이 검증 가능한 테스트를 구축한다.

**Non-Goals:**
- 실제 AI 기반 고도 추천 품질 최적화
- 프론트엔드 UI 완성도 개선
- 구독/결제 플랜 연동

## Decisions

1. 도메인 판정과 추천 로직을 `domainamer/services/domain_recommender.py`로 분리한다.
- 이유: 뷰/URL과 독립된 순수 로직으로 만들어 테스트성을 높이고 재사용 경로를 확보한다.
- 대안: `views.py`에 직접 구현. 단기 구현은 빠르지만 테스트 경계가 약해져 이후 유지보수 비용이 커진다.

2. 응답 계약은 리스트 단위 JSON 구조로 표준화한다.
- 형식: `{"results": [{"candidate": str, "status": "available|unavailable|duplicate", "reason": str|null, "alternatives": [str]}]}`
- 이유: HTMX partial 렌더링과 API 소비를 동시에 지원하기 쉽다.
- 대안: 단일 문자열 응답/템플릿 전용 컨텍스트. 소비자 확장성이 떨어진다.

3. 가용성 체크는 추상 함수(체커 주입)로 설계한다.
- 이유: 초기에는 스텁 체커로 빠르게 동작시키고, 이후 WHOIS 구현으로 교체 가능하다.
- 대안: WHOIS 직접 호출 고정. 테스트에서 외부 네트워크 의존이 생긴다.

4. 추천 생성 규칙은 결정적 규칙 기반으로 시작한다.
- 규칙: 숫자/짧은 접미사(`go`, `lab`, `hq`, `app`)를 붙여 최대 N개 추천.
- 이유: 초기 단계에서 예측 가능한 결과와 간단한 테스트가 가능하다.
- 대안: LLM 기반 추천. 품질은 높을 수 있으나 비용/지연/비결정성이 증가한다.

## Risks / Trade-offs

- [외부 WHOIS 연동 지연] -> Mitigation: 인터페이스 분리 후 스텁 체커 기본값 유지
- [추천 품질 한계] -> Mitigation: 규칙 기반으로 시작하고 향후 랭킹 전략 교체 가능 구조 유지
- [중복 정규화 과소/과다 판정] -> Mitigation: 정규화 규칙을 유틸 함수로 고정하고 테스트 케이스 확장
- [응답 포맷 변경으로 인한 호환성 이슈] -> Mitigation: 새 엔드포인트(`domainamer/recommend/`)로 시작해 점진 전환

## Migration Plan

1. 서비스 모듈 및 유틸 함수 추가
2. `POST /domainamer/recommend/` 엔드포인트 추가
3. 단위/통합 테스트 추가
4. 기존 생성 플로우와 연결(추후 단계)

Rollback: URL 등록 제거 및 서비스 모듈 참조 제거로 즉시 원복 가능.
